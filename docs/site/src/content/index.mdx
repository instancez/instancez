Ultrabase turns a single YAML file into a production-ready REST API with authentication, file storage, event-driven webhooks, and custom SQL functions — all powered by PostgreSQL.

## Why Ultrabase?

- **One file, full backend** — tables, auth, storage, events, and functions all declared in YAML
- **No vendor lock-in** — your config is a YAML file, your data lives in Postgres
- **Version-controllable** — the YAML file is the source of truth, not a dashboard
- **PostgREST-compatible** — familiar query interface with filtering, pagination, and embedding
- **Single binary** — no Docker Compose with 12 services, just one Go binary

<Callout variant="tip" title="Who is Ultrabase for?">
  Solo developers and small teams who want a backend without writing backend code.
  Agencies shipping multiple client projects. Prototyping teams that need a real backend in minutes.
</Callout>

## Quick example

```yaml
# ultrabase.yaml
project:
  name: my-app

tables:
  todos:
    fields:
      id: { type: bigserial, primary_key: true }
      title: { type: text, required: true }
      done: { type: boolean, default: "false" }
      created_at: { type: timestamptz, default: "now()" }

auth:
  jwt_expiry: 15m
  email:
    verify_email: true
```

```bash
ultrabase dev
```

That gives you a REST API with auth, automatic migrations, and a Swagger UI at `/docs`.

## How it compares

| | Ultrabase | Supabase | Firebase | Hasura |
|---|---|---|---|---|
| Config | YAML | Dashboard/SQL | Dashboard/JSON | Console/metadata |
| Query | PostgREST | PostgREST | Proprietary SDK | GraphQL |
| Auth | Built-in | GoTrue | Built-in | External |
| Custom logic | SQL in YAML | Edge functions | Cloud functions | Actions |
| Self-hosted | Single binary | Docker (many services) | No | Docker |
| Source of truth | YAML file | Database | Dashboard | Metadata files |
