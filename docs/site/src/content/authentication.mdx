Ultrabase has built-in authentication with email/password, OAuth providers, MFA, and passwordless login. Auth is configured entirely in YAML — no external service required. The auth API is GoTrue-compatible, so the `@supabase/supabase-js` client works without changes.

## Enabling auth

```yaml
auth:
  jwt_expiry: 15m
  refresh_tokens: true
  refresh_token_expiry: 7d
```

This gives you auth endpoints:

| Endpoint | Method | Description |
|---|---|---|
| `/auth/v1/signup` | POST | Create account |
| `/auth/v1/token` | POST | Sign in (email/password or refresh token) |
| `/auth/v1/user` | GET | Get current user |
| `/auth/v1/user` | PUT | Update user profile |
| `/auth/v1/logout` | POST | Invalidate refresh token |
| `/auth/v1/recover` | POST | Initiate password reset |
| `/auth/v1/verify` | POST | Verify email or reset token |
| `/auth/v1/otp` | POST | Passwordless OTP (magic link / 6-digit code) |
| `/auth/v1/settings` | GET | Public auth configuration |

### MFA endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/auth/v1/factors` | GET | List enrolled MFA factors |
| `/auth/v1/factors` | POST | Enroll new TOTP factor |
| `/auth/v1/factors/:id` | DELETE | Unenroll MFA factor |
| `/auth/v1/factors/:id/challenge` | POST | Create MFA challenge |
| `/auth/v1/factors/:id/verify` | POST | Verify MFA challenge (upgrades JWT to `aal2`) |

### OAuth endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/auth/v1/authorize` | GET | Start OAuth flow |
| `/auth/v1/callback/google` | GET | Google OAuth callback |
| `/auth/v1/callback/github` | GET | GitHub OAuth callback |

### Admin endpoints

These require the `ULTRABASE_ADMIN_KEY` as the bearer token:

| Endpoint | Method | Description |
|---|---|---|
| `/auth/v1/admin/users` | POST | Create user |
| `/auth/v1/admin/users` | GET | List users |
| `/auth/v1/admin/users/:uid` | GET | Get user by ID |
| `/auth/v1/admin/users/:uid` | PUT | Update user |
| `/auth/v1/admin/users/:uid` | DELETE | Delete user |
| `/auth/v1/admin/generate_link` | POST | Generate sign-in or reset link |
| `/auth/v1/invite` | POST | Send invite email |

## Email verification

```yaml
auth:
  email:
    verify_email: true
    templates:
      verify:
        subject: "Verify your email"
        body: |
          Hi {{data.display_name}},
          Click here to verify: {{link}}
      reset:
        subject: "Reset your password"
        body: |
          Click here to reset your password: {{link}}
```

<Callout variant="info" title="Email provider required">
  Email verification requires an email provider configured under `providers.email`.
  Supported: `resend`, `sendgrid`, `ses`.
</Callout>

## Custom user fields

Custom fields are added to the `users` table under `tables`, not under `auth`. The migrator automatically adds core columns (id, email, password_hash, etc.), so you only declare your custom fields:

```yaml
tables:
  users:
    fields:
      - name: display_name
        type: text
      - name: avatar_url
        type: text
      - name: role
        type: text
        default: "'member'"
      - name: org_id
        foreign_key:
          references: orgs.id
```

These fields are included in the JWT claims and returned by `/auth/v1/user`.

## OAuth providers

### Google

```yaml
auth:
  google:
    client_id: ${GOOGLE_CLIENT_ID}
    client_secret: ${GOOGLE_CLIENT_SECRET}
    redirect_url: http://localhost:3000/auth/callback
```

### GitHub

```yaml
auth:
  github:
    client_id: ${GITHUB_CLIENT_ID}
    client_secret: ${GITHUB_CLIENT_SECRET}
    redirect_url: http://localhost:3000/auth/callback
```

<Callout variant="tip" title="Environment variables">
  Use `${ENV_VAR}` syntax in your YAML. Ultrabase interpolates environment variables at startup,
  so secrets never live in the config file.
</Callout>

## Using auth in RLS

The `auth.uid()` function returns the authenticated user's ID. Use it in RLS policies:

```yaml
tables:
  posts:
    rls:
      - operations: [select, insert, update, delete]
        check: "author_id = auth.uid()"
```

The JWT also carries custom fields, accessible via `auth.jwt() ->> 'field_name'`:

```yaml
rls:
  - operations: [select]
    check: "org_id = (auth.jwt() ->> 'org_id')::bigint"
```
