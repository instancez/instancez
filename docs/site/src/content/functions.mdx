Functions let you define PostgreSQL stored procedures in YAML and expose them as REST endpoints at `/rest/v1/rpc/<name>`, fully compatible with `supabase-js .rpc()`.

## Defining a function

```yaml
functions:
  get_team_members:
    description: Get all members for a team
    auth_required: true
    language: plpgsql
    volatility: stable
    security: invoker
    args:
      - name: team_id
        type: bigint
        required: true
    returns:
      type: "setof team_members"
    body: |
      BEGIN
        RETURN QUERY
        SELECT * FROM team_members
        WHERE team_members.team_id = get_team_members.team_id;
      END;
```

This creates a real PostgreSQL function and exposes it at `/rest/v1/rpc/get_team_members`.

## Function options

| Option | Description |
|---|---|
| `description` | Human-readable description (shown in OpenAPI docs) |
| `auth_required` | If true, requires a valid JWT to call |
| `language` | `sql` or `plpgsql` (default: `plpgsql`) |
| `volatility` | `volatile`, `stable`, or `immutable` (default: `volatile`) |
| `security` | `invoker` or `definer` (default: `invoker`) |
| `args` | Ordered list of function arguments |
| `returns.type` | Raw PostgreSQL return type (e.g., `int`, `void`, `setof posts`) |
| `body` | The function body (PL/pgSQL or SQL) |

## Arguments

Arguments are declared as an ordered list. Each argument has a name, type, and optional default:

```yaml
args:
  - name: org_id
    type: bigint
    required: true
  - name: status
    type: text
    required: false
  - name: max_results
    type: integer
    default: 50
```

## Return types

The `returns.type` field accepts any valid PostgreSQL return type:

| Type | Description |
|---|---|
| `void` | No return value (HTTP 204) |
| `int`, `text`, etc. | Scalar value |
| `setof <table>` | Array of rows from a table |
| `record` | Single anonymous record |
| `table(id int, name text)` | Inline table definition |

## HTTP methods

<Callout variant="info" title="GET vs POST">
  Functions with `stable` or `immutable` volatility can be called via GET.
  `volatile` functions require POST.
</Callout>

## SQL language functions

For simple queries, use `language: sql`:

```yaml
functions:
  add_numbers:
    language: sql
    volatility: immutable
    args:
      - name: a
        type: int
        required: true
      - name: b
        type: int
        required: true
    returns:
      type: int
    body: "SELECT a + b"
```

## PL/pgSQL functions

For procedural logic, use `language: plpgsql`:

```yaml
functions:
  archive_old_tasks:
    auth_required: true
    language: plpgsql
    volatility: volatile
    returns:
      type: void
    body: |
      BEGIN
        UPDATE tasks SET archived = true
        WHERE completed_at < now() - interval '90 days';
      END;
```

## Calling functions

Functions are called via the standard PostgREST RPC interface:

```bash
# POST (works for all functions)
curl -X POST http://localhost:8080/rest/v1/rpc/add_numbers \
  -H "Content-Type: application/json" \
  -d '{"a": 1, "b": 2}'

# GET (only for stable/immutable functions)
curl "http://localhost:8080/rest/v1/rpc/add_numbers?a=1&b=2"
```

With `supabase-js`:

```js
const { data, error } = await supabase.rpc('add_numbers', { a: 1, b: 2 })
```

<Callout variant="warning" title="No RLS enforcement">
  Functions bypass RLS. They run with the privileges of the database role,
  so validate authorization in the SQL or via the `auth_required` flag.
</Callout>
