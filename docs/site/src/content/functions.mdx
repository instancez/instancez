Ultrabase supports two kinds of functions:

- **`rpc:`** — PostgreSQL stored procedures exposed at `/rest/v1/rpc/<name>`, compatible with `supabase-js .rpc()`.
- **`functions:`** — JavaScript (ESM) handlers served at `/functions/v1/<name>`, invoked via `supabase-js .functions.invoke()`.

---

## Postgres functions (`rpc:`)

Define PostgreSQL stored procedures in YAML and expose them as REST endpoints.

```yaml
rpc:
  get_team_members:
    description: Get all members for a team
    auth_required: true
    language: plpgsql
    volatility: stable
    security: invoker
    args:
      - name: team_id
        type: bigint
        required: true
    returns:
      type: "setof team_members"
    body: |
      BEGIN
        RETURN QUERY
        SELECT * FROM team_members
        WHERE team_members.team_id = get_team_members.team_id;
      END;
```

This creates a real PostgreSQL function and exposes it at `/rest/v1/rpc/get_team_members`.

### Function options

| Option | Description |
|---|---|
| `description` | Human-readable description (shown in OpenAPI docs) |
| `auth_required` | If true, requires a valid JWT to call |
| `language` | `sql` or `plpgsql` (default: `plpgsql`) |
| `volatility` | `volatile`, `stable`, or `immutable` (default: `volatile`) |
| `security` | `invoker` or `definer` (default: `invoker`) |
| `args` | Ordered list of function arguments |
| `returns.type` | Raw PostgreSQL return type (e.g., `int`, `void`, `setof posts`) |
| `body` | The function body (PL/pgSQL or SQL) |

### Arguments

Arguments are declared as an ordered list. Each argument has a name, type, and optional default:

```yaml
args:
  - name: org_id
    type: bigint
    required: true
  - name: status
    type: text
    required: false
  - name: max_results
    type: integer
    default: 50
```

### Return types

The `returns.type` field accepts any valid PostgreSQL return type:

| Type | Description |
|---|---|
| `void` | No return value (HTTP 204) |
| `int`, `text`, etc. | Scalar value |
| `setof <table>` | Array of rows from a table |
| `record` | Single anonymous record |
| `table(id int, name text)` | Inline table definition |

### HTTP methods

<Callout variant="info" title="GET vs POST">
  Functions with `stable` or `immutable` volatility can be called via GET.
  `volatile` functions require POST.
</Callout>

### SQL language functions

For simple queries, use `language: sql`:

```yaml
rpc:
  add_numbers:
    language: sql
    volatility: immutable
    args:
      - name: a
        type: int
        required: true
      - name: b
        type: int
        required: true
    returns:
      type: int
    body: "SELECT a + b"
```

### PL/pgSQL functions

For procedural logic, use `language: plpgsql`:

```yaml
rpc:
  archive_old_tasks:
    auth_required: true
    language: plpgsql
    volatility: volatile
    returns:
      type: void
    body: |
      BEGIN
        UPDATE tasks SET archived = true
        WHERE completed_at < now() - interval '90 days';
      END;
```

### Calling Postgres functions

Functions are called via the standard PostgREST RPC interface:

```bash
# POST (works for all functions)
curl -X POST http://localhost:8080/rest/v1/rpc/add_numbers \
  -H "Content-Type: application/json" \
  -d '{"a": 1, "b": 2}'

# GET (only for stable/immutable functions)
curl "http://localhost:8080/rest/v1/rpc/add_numbers?a=1&b=2"
```

With `supabase-js`:

```js
const { data, error } = await supabase.rpc('add_numbers', { a: 1, b: 2 })
```

<Callout variant="warning" title="No RLS enforcement">
  Functions bypass RLS. They run with the privileges of the database role,
  so validate authorization in the SQL or via the `auth_required` flag.
</Callout>

## Code functions (`functions:`)

Code functions are JavaScript (ESM) HTTP handlers served at `/functions/v1/<name>`. They run in Node.js worker processes and have access to the caller's Postgres session via a `@supabase/supabase-js` client.

### Authoring

Create a file with an ESM default export:

```js
// functions/hello.js
export default async function handler(req, ctx) {
  const name = req.body?.name ?? "world";
  ctx.log.info("hello called", { name });
  return {
    status: 200,
    body: { hello: name },
  };
}
```

The `handler` receives two arguments:

**`req`** — the incoming HTTP request:

| Property | Description |
|---|---|
| `req.method` | HTTP method (`"GET"`, `"POST"`, …) |
| `req.path` | URL path |
| `req.query` | Query parameters as `{ key: firstValue }` |
| `req.headers` | Request headers (lowercased keys) |
| `req.body` | Parsed body — object for `application/json`, string otherwise |

**`ctx`** — the injected context:

| Property | Description |
|---|---|
| `ctx.supabase` | `@supabase/supabase-js` client carrying the caller's JWT — RLS applies |
| `ctx.serviceClient` | `@supabase/supabase-js` client with a service_role token (BYPASSRLS) |
| `ctx.claims` | Decoded JWT claims (`{ sub, role, email, … }`) or `null` for anonymous callers |
| `ctx.env` | Resolved secrets from the function's `env:` block |
| `ctx.log` | Structured logger — `.info(msg, fields?)`, `.warn`, `.error`, `.debug` |
| `ctx.signal` | `AbortSignal` that fires when the per-request timeout elapses |

The handler must return an object with `status` (HTTP status code) and `body` (serialized as JSON when it is an object).

### Declaring a function

```yaml
functions:
  hello:
    runtime: node          # required; "node" is the only supported runtime
    file: functions/hello.js   # path relative to ultrabase.yaml
    auth_required: false   # when true, ultrabase returns 401 for unauthenticated callers before invoking
    timeout: 30s           # per-request timeout (default: 30s)
    env:
      API_KEY: "${ULTRA_ENV_MY_API_KEY}"  # reference a secret from ULTRA_ENV_*
      STATIC_VAL: "literal"              # plain string (not from env)
```

### Secrets (`env:` and `ULTRA_ENV_*`)

Secrets are injected from the `ULTRA_ENV_*` namespace, not from arbitrary process environment variables. This prevents accidental leakage of host secrets (AWS credentials, database URLs, etc.) into function workers.

1. Set `ULTRA_ENV_MY_API_KEY=secret` in `.env` / `.<mode>.env` or your process environment.
2. Reference it in the `env:` block as `"${ULTRA_ENV_MY_API_KEY}"`.
3. Read it in the handler as `ctx.env.API_KEY`.

At startup, the runtime resolves every `${ULTRA_ENV_*}` reference and fails early if a referenced variable is missing. Secrets are passed to workers via an internal per-request header — they are never written to the worker's process environment.

### Lifecycle: dev → deploy → serve

| Mode | Behavior |
|---|---|
| `ultra dev` | Runs functions directly from the local `functions/` directory. Installs npm dependencies on the fly if `package.json` is present. Hot-reloads on YAML change. |
| `ultra deploy` | Vendors dependencies (`npm ci`), packs a tarball containing source + `node_modules`, uploads it to the configured bundle destination, and records the bundle pointer in the config. |
| `ultra serve` | Consumes the pre-built bundle produced by `deploy`. Never builds — the bundle must already exist. |

### Calling code functions

```bash
curl -X POST http://localhost:8080/functions/v1/hello \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"name": "ultrabase"}'
```

With `supabase-js`:

```js
const { data, error } = await supabase.functions.invoke('hello', {
  body: { name: 'ultrabase' },
})
```
