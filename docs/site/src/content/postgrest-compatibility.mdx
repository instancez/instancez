Ultrabase implements the Supabase HTTP APIs so the standard `@supabase/supabase-js` client works without client-side changes for the features listed below. This page documents the current state across all three surfaces: **Auth**, **REST** (PostgREST), and **Storage**.

```js
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(ULTRABASE_URL, ULTRABASE_ANON_KEY)
```

## REST (`/rest/v1/`)

Ultrabase exposes a PostgREST-compatible HTTP surface. The supabase-js `.from()`, `.rpc()`, and query builder methods work without changes.

### At a glance

| Area | Status | Notes |
|---|---|---|
| **CRUD** — `GET`/`POST`/`PATCH`/`DELETE`, bulk insert + `columns=`, upsert (`on_conflict=`), `?select=` on mutations | ✅ Full | |
| **`Prefer` headers** — `return=representation`/`minimal`/`headers-only`, `tx=rollback`, `count=exact`/`planned`/`estimated`, `max-affected=N`, `handling=strict` | ✅ Full | |
| **Filter operators** — `eq`/`neq`/`gt[e]`/`lt[e]`, `like`/`ilike` (incl. `all`/`any` variants), `is`, `in`, `cs`/`cd`/`ov`, `match`/`imatch`, `isdistinct` | ✅ Full | |
| **Range operators** — `sl`, `sr`, `nxl`, `nxr`, `adj` | ✅ Full | |
| **Logical trees** — `and=`, `or=`, `not.` prefixes, nested | ✅ Full | |
| **Resource embedding** — belongs-to, has-many, `!inner`, nested scope, spread (`...`), FK-hint disambiguation | ✅ Full | |
| **Content types** — `application/json`, `application/vnd.pgrst.object+json`, `text/csv` | ✅ Full | |
| **JSONB path access** — `metadata->>key` in filters and select | ✅ Full | |
| **Full-text search** — `fts`, `plfts`, `phfts`, `wfts` | ✅ Full | |
| **RPC** — `.rpc()` with scalar/SETOF/void returns, chaining, `params=single-object` | ✅ Full | |
| **Aggregates** — `count()`, `avg()`, `sum()`, `min()`, `max()`, aliases, GROUP BY, HAVING | ✅ Full | |
| **Error envelope** — `{code, message, details, hint}` with SQLSTATE + `PGRST*` codes | ✅ Full | |
| **EXPLAIN** — `Accept: application/vnd.pgrst.plan+json` returns query plan | ✅ | JSON and text formats |
| **GeoJSON** — `Accept: application/geo+json` wraps geometry columns as FeatureCollection | ✅ | Requires PostGIS + geometry/geography column |
| **Multi-schema** — `Accept-Profile`/`Content-Profile` headers | ✅ | Tables can declare `schema:` in YAML; `SET LOCAL search_path` per request |
| **DB timeout** — `Prefer: statement-timeout=N` (ms) | ✅ | Capped at configured `server.timeouts.db_query` |
| **Request cancellation** (`abortSignal`) | ✅ | Client-side only; HTTP context propagates to pgx |
| Views as endpoints | ❌ Missing | |
| Computed columns from `function(row)` | ❌ Missing | |

### Error codes

Constraint violations forward real SQLSTATEs:

| SQLSTATE | HTTP | Kind |
|---|---|---|
| `23505` | 409 | unique_violation |
| `23503` | 422 | foreign_key_violation |
| `23502` | 422 | not_null_violation |
| `23514` | 422 | check_violation |
| `22P02` | 422 | invalid_text_representation |
| `42501` | 403 | insufficient_privilege / RLS denial |
| `42P01` | 404 | undefined_table |
| `42703` | 404 | undefined_column |

API-level errors use `PGRST*` codes: `PGRST100` (parse), `PGRST106` (schema), `PGRST116` (single-row), `PGRST122` (strict), `PGRST124` (max-affected), `PGRST202` (unknown function), `PGRST301` (auth).

## Storage (`/storage/v1/`)

Ultrabase implements the Supabase Storage API so `supabase.storage` methods work out of the box.

### Fully compatible methods

| Method | Status | Notes |
|---|---|---|
| `storage.listBuckets()` | ✅ | |
| `storage.getBucket(id)` | ✅ | |
| `storage.from(b).list(folder)` | ✅ | |
| `storage.from(b).listV2(folder)` | ✅ | Cursor-based pagination, `with_delimiter` for folders |
| `storage.from(b).info(path)` | ✅ | |
| `storage.from(b).exists(path)` | ✅ | |
| `storage.from(b).remove([paths])` | ✅ | |
| `storage.from(b).move(from, to)` | ✅ | Server-side copy + delete |
| `storage.from(b).copy(from, to)` | ✅ | Server-side copy |
| `storage.from(b).createSignedUrl(path, exp)` | ✅ | Returns direct S3 URL |
| `storage.from(b).createSignedUrls(paths, exp)` | ✅ | Batch version |
| `storage.from(b).createSignedUploadUrl(path)` | ✅ | |
| `storage.from(b).getPublicUrl(path)` | ✅ | Client-side URL construction |
| `storage.from(b).upload(path, file)` | ✅ | Proxied, capped at bucket `max_size` |
| `storage.from(b).update(path, file)` | ✅ | Proxied, capped at bucket `max_size` |
| `storage.from(b).download(path)` | ✅ | Supports image transforms via query params |
| `storage.from(b).uploadToSignedUrl(path, token, file)` | ✅ | Proxied, capped at bucket `max_size` |
| `storage.from(b).emptyBucket()` | ✅ | |
| Image transforms | ✅ | `?width=&height=&resize=&quality=&format=` on download |
| `storage.createBucket()` | ❌ | Buckets are defined in YAML |
| `storage.updateBucket()` | ❌ | |
| `storage.deleteBucket()` | ❌ | |
| Resumable uploads (TUS) | ❌ | |
| Vector storage | ❌ | |

<Callout variant="warning" title="Serverless payload limits">
  The 4 proxied methods (upload, update, download, uploadToSignedUrl) stream file bytes through the server. In serverless environments, these are constrained by request/response payload limits (e.g., 6MB on AWS Lambda). For large files, use the serverless-friendly presigned URL endpoints described on the [Storage](/docs/storage) page.
</Callout>

## Testing compatibility

The integration suite runs both `postgrest-go` and `@supabase/supabase-js` against a live Ultrabase server with a real Postgres instance:

- **postgrest-go conformance** — `internal/adapter/http/pgrupstream/conformance_test.go`: filters, transforms, embedding, mutations, error envelope
- **supabase-js harness** — `test/integration/supabase-js/run.mjs`: auth flows (signup, signin, refresh, MFA, anonymous, OTP, PKCE, ID token, identity linking, admin ops), REST operations (CRUD, filters, embeds, upsert, RPC, explain), and storage operations (all endpoints + image transforms + serverless presigned URLs)

```sh
go test -tags=integration ./internal/adapter/http/...
```
