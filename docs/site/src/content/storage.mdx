Ultrabase provides file storage with per-bucket configuration, MIME type filtering, and row-level security.

## Defining a bucket

```yaml
storage:
  avatars:
    max_size: 5MB
    types: ["image/jpeg", "image/png", "image/webp"]
    public: true

  documents:
    max_size: 50MB
    types: ["application/pdf"]
    public: false
    rls:
      - operations: [select, insert, delete]
        check: "owner_id = auth.uid()"
```

## Configuration options

| Option | Description |
|---|---|
| `max_size` | Maximum file size (e.g., `5MB`, `100KB`) |
| `types` | Allowed MIME types, supports wildcards (`image/*`) |
| `public` | If true, files are accessible without authentication |
| `rls` | Row-level security policies (same syntax as tables) |

## Storage providers

Configure where files are stored under `providers.storage`:

```yaml
providers:
  storage:
    type: s3  # or: minio, local
```

<Callout variant="info" title="Provider configuration">
  Provider credentials are configured via environment variables (e.g., `S3_BUCKET`, `S3_REGION`).
  The `local` provider stores files on disk — useful for development.
</Callout>

## Using storage with supabase-js

The standard supabase-js storage client works with Ultrabase. See the [Supabase Compatibility](/docs/postgrest-compatibility) page for the full method-by-method breakdown.

```js
const { data, error } = await supabase.storage
  .from('avatars')
  .upload('profile.png', file)
```

<Callout variant="warning" title="Serverless payload limits">
  Upload, update, download, and uploadToSignedUrl proxy file bytes through the server, capped at the bucket's `max_size` (or 50MB default). In serverless environments this is constrained by payload limits. For large files, use the serverless-friendly endpoints below.
</Callout>

## Serverless-friendly endpoints

For files that exceed the proxy size limit — or in serverless environments where you can't stream large bodies through the server — use the serverless-friendly presigned URL endpoints to upload and download directly to/from S3.

### Upload via presigned URL

```js
// 1. Get a presigned upload URL from ultrabase
const resp = await fetch(`${ULTRABASE_URL}/api/storage/avatars/sign`, {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${accessToken}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    content_type: 'image/png',
    size: file.size,
  }),
});
const { id, upload_url } = await resp.json();

// 2. Upload directly to S3
await fetch(upload_url, {
  method: 'PUT',
  headers: { 'Content-Type': 'image/png' },
  body: file,
});
```

The `upload_url` is a presigned S3 URL. File bytes go straight from the client to S3 — they never touch the ultrabase server.

### Download via presigned URL

```js
// 1. Get a presigned download URL
const resp = await fetch(
  `${ULTRABASE_URL}/api/storage/avatars/${objectId}`,
  {
    headers: { 'Authorization': `Bearer ${accessToken}` },
  }
);
const { url } = await resp.json();

// 2. Download directly from S3
const file = await fetch(url);
```

### When to use which approach

| Approach | Use when |
|---|---|
| `supabase.storage.from().upload()` | Files under the bucket's `max_size`, running on a traditional server |
| Serverless-friendly (`/api/storage/.../sign`) | Large files, serverless environments, or when you need direct-to-S3 transfers |
| `supabase.storage.from().createSignedUrl()` | Generating temporary download links to share with users |

<Callout variant="info" title="Ultrabase-specific endpoints">
  The serverless-friendly endpoints at `/api/storage/` are ultrabase-specific and not part of the supabase-js client. You call them with plain `fetch`.
</Callout>

## API reference

### supabase-js compatible endpoints (`/storage/v1/`)

| Endpoint | Method | Description |
|---|---|---|
| `/storage/v1/bucket` | GET | List buckets |
| `/storage/v1/bucket/:id` | GET | Get bucket details |
| `/storage/v1/bucket/:id/empty` | POST | Empty a bucket |
| `/storage/v1/object/:bucket/:path` | POST | Upload a file |
| `/storage/v1/object/:bucket/:path` | PUT | Update a file |
| `/storage/v1/object/:bucket/:path` | HEAD | Check if file exists |
| `/storage/v1/object/authenticated/:bucket/:path` | GET | Download (authenticated) |
| `/storage/v1/object/public/:bucket/:path` | GET | Download (public buckets) |
| `/storage/v1/object/list/:bucket` | POST | List objects |
| `/storage/v1/object/info/authenticated/:bucket/:path` | GET | Object metadata |
| `/storage/v1/object/:bucket` | DELETE | Remove objects (paths in body) |
| `/storage/v1/object/move` | POST | Move an object |
| `/storage/v1/object/copy` | POST | Copy an object |
| `/storage/v1/object/sign/:bucket/:path` | POST | Create signed download URL |
| `/storage/v1/object/sign/:bucket` | POST | Batch signed download URLs |
| `/storage/v1/object/upload/sign/:bucket/:path` | POST | Create signed upload token |
| `/storage/v1/object/upload/sign/:bucket/:path` | PUT | Upload with signed token |

### Serverless-friendly endpoints (`/api/storage/`)

| Endpoint | Method | Description |
|---|---|---|
| `/api/storage/:bucket/sign` | POST | Get presigned upload URL (direct to S3) |
| `/api/storage/:bucket/:id` | GET | Get presigned download URL (direct from S3) |
| `/api/storage/:bucket/:id` | DELETE | Delete a file |
