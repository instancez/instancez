# Docs Site Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the instancez docs site at `docs/site/` using Starlight (Astro) with a custom marketing landing page, four content sections (Quick Start, Build, Deploy, API Reference), and GitHub Pages deployment.

**Architecture:** Starlight project lives at `docs/site/` inside the main repo. A custom `src/pages/index.astro` completely replaces Starlight's default home page with the approved marketing landing page. All 18 doc pages live in `src/content/docs/` as Markdown files with Starlight frontmatter. A GitHub Actions workflow builds and deploys to GitHub Pages on push to `main`.

**Tech Stack:** Astro 5.x, @astrojs/starlight 0.34.x, expressive-code with `material-theme-darker` Shiki theme, Pagefind (bundled), GitHub Actions + GitHub Pages.

## Task 5: GitHub Actions CI/CD

**Files:**
- Create: `.github/workflows/docs.yml`

- [ ] **Step 1: Create `.github/workflows/docs.yml`**

```yaml
name: Deploy docs

on:
  push:
    branches: [main]
    paths:
      - 'docs/site/**'
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: docs/site/package-lock.json

      - name: Install dependencies
        run: npm ci
        working-directory: docs/site

      - name: Build
        run: npm run build
        working-directory: docs/site

      - name: Upload Pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: docs/site/dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

```markdown
---
title: RLS Policies
description: Row-level security is the only authorization layer in instancez. All access decisions are Postgres policies.
---

instancez has no HTTP-level RBAC. Every access decision is a Postgres RLS policy declared in `instancez.yaml`. The middleware validates the JWT, sets the Postgres role, and RLS does the rest.

## Policy syntax

```yaml
tables:
  posts:
    rls:
      - operations: [select]
        check: "true"                        # allow all reads
      - operations: [insert, update, delete]
        check: "user_id = auth.uid()"        # owner only
```

`operations` accepts any combination of `select`, `insert`, `update`, `delete`. `check` is a SQL boolean expression — the same syntax as a Postgres `USING` clause.

Multiple policies combine with OR: a row is accessible if any policy passes.

## auth.uid() and auth.is_authenticated()

These are Postgres functions installed by instancez:

| Function | Returns | Condition |
|---|---|---|
| `auth.uid()` | `uuid` or `null` | The `sub` claim from the JWT, or `null` for anonymous/no-auth requests |
| `auth.is_authenticated()` | `boolean` | `true` when the caller has a valid JWT with `role = 'authenticated'` |

They read session GUCs set by the request middleware — they are not application-layer checks.

## Common patterns

**Public read, owner write:**
```yaml
rls:
  - operations: [select]
    check: "true"
  - operations: [insert, update, delete]
    check: "user_id = auth.uid()"
```

**Signed-in users only:**
```yaml
rls:
  - operations: [select, insert]
    check: "auth.is_authenticated()"
```

**Private (owner only, all operations):**
```yaml
rls:
  - operations: [select, insert, update, delete]
    check: "user_id = auth.uid()"
```

**Admin bypass via service role:**

The `service_role` has `BYPASSRLS` — it skips all policies. Use the admin key in code functions via `ctx.serviceClient` when you need to write rows that RLS would otherwise block.

## How roles are assigned

The JWT's `role` claim determines the Postgres role:

| JWT `role` claim | Postgres role | BYPASSRLS |
|---|---|---|
| absent / invalid JWT | `anon` | No |
| `authenticated` | `authenticated` | No |
| `service_role` | `service_role` | Yes |

The Postgres role identifiers are configurable via `INSTANCEZ_DB_*_ROLE` env vars, but the JWT wire values (`anon`, `authenticated`, `service_role`) are fixed for supabase-js compatibility.
